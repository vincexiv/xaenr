We are building a one-shot image recognition algorithm that is explainable by design. If you are interested in contributing to this project, please read along.

So far, the first phase is almost done. In this part, I mainly focused on getting the algorithm to work on small dataset such as MNIST digits and the random doodles from the test it page . If you are curious about how to submit samples for test, visit this page.

The next phase is scaling the project's capability such that it would work with bigger images.

In this discussion, I will be taking you through the classes, and the methods that you could use to experiment on what I have done and assuming you will be interested in contributing, add or modify a few things to make it more robust, efficient, and able to handle bigger images

The content provided here is broken down into four parts;

  1. Get a two dimensional array to represent the sample
  2. Transform the element values in the array
  3. Generate an array of Perspectives corresponding to the original sample
  4. Compare the arrays of perspectives. This is analogous to comparing the two samples

Throughout this discussion, I'll be using terms such as "sample", "element" a lot. Here's their definition;

  • sample: a two dimensional array of numbers. Could be obtained from a picture or any other source
  • element: a point in the two dimensional array. An example of an element is (1, 1)
  • angle: I have used angle and orientation interchangeably. Whenever you see "angle" think of orientation. The "angle" as used in this work only shows how horizontal or vertical an element is. It makes no distinction of an angle being $45^\circ$ or $135^\circ$. Both are treated the same. I did that to make the algorithm robust with respect to image rotation

The Perspective object also offers us a method to compare one perspective with another. So far we have created two perspectives from the sample we have. To recap, here are the two perspectives

In [50]:
perspective1.showElement()

comparison is done by comparing all elements in one perspective against another to create a two dimensional array of cost that show how much distance the two elements have between themselves. Please note that we don't use the Euclidean or any of the conventional methods of calculating distance. Rather, the calculation is informed by what I discussed here. Our goal is to force close elements in the original sample to be mapped to regions in another sample in such a way that they are, again, close to each other. I'll use a small example here.

Assume that our first array of elements is [1, 2, 3, 4], and the other one is [2, 3, 4, 5]. We want to match the items in the first array with that of the second such that no element has been mapped onto more than one element in the other array. For instance, if we map 1 in the first array to 2 in the second, we aren't allowed to map 1 to any other number in the second. We also need to ensure that the sum of difference is the least it could be.

First we will create a two dimensional array that shows all the distances. I'll represent this using a pandas DataFrame for clarity's sake

In [51]:
result = {}

for i in [1, 2, 3, 4]:
    result[i] = {}
    for j in [2, 3, 4, 5]:
        result[i][j] = abs(i - j)


pd.DataFrame(result)
Out[51]:
1 2 3 4
2 1 0 1 2
3 2 1 0 1
4 3 2 1 0
5 4 3 2 1

We can see that the distance between 1 and 2 is 1, the distance between 1 and 3 is 2, etc. Thus, if we map 1 to 3, our cost = 2. We now remain with [2, 3, 4] from the first array and [2, 4, 5] from the second array. From this array, we could again map 2 from the first array to 4 in the second. The distance between them is 2, and thus our total cost at this point = 4 (2 + 2). We can go on like that until each item from the first has a "partner" from the second then calculate the total cost.

Since we need the total cost to be the least it could be, we won't just be doing guess work like we have done above

The solution to the above problem is by mapping 1 from the first array to 2 in the second, 2 from the first to 3 in the second, etc. Generally, the optimal mapping is as follows;

$2 \rightarrow 2 $
$3 \rightarrow 3 $
$4 \rightarrow 4 $
$5 \rightarrow 1 $

The mapping above can easily be obtained by simply feeding the table of distances I showed previously to the Hungarian algorithm. Given the mapping above, the total cost = 4, which is the optimal solution

Another solution to the above problem is;

$1 \rightarrow 2 $
$2 \rightarrow 3 $
$3 \rightarrow 4 $
$4 \rightarrow 5 $

They can both be interpreted differently. For the first, the $5 \arrowright 1$ considered not a match, while the rest are perfect match. As for the second scenario, the second array is simply the same as array 1 with a transformation applied

Any different mapping and the total cost would likely be higher than 4

Now think of the array of numbers as array of perspectives; $[p_1, p_2, ..., p_n]$. Thus, the distance between elements is actually the distance between the perspectives, since each of the perspective can be represented by the element that created it. The table of distance between perspectives can be obtianed from the Compare object that is returned by the Perspective.compare() method that I will be demonstrating shortly. We can then simply feed the table into the Hungarian algorithm to get the optimal mapping.

We can determine how close a sample is similar to another by checking how close elements that were neighbors in the original sample are in the new sample. In the case of the first comparison, we can see that for all elements their neighboring elements in the original sample were also their neighboring elements in the other sample after mapping. We can calculate the average distance of neighboring elements after mapping. The smaller this value is, the better the match

Another awesome information provided by the Compare object is the ability to see the transformation matrix of the matches. If it considers one image to be a match for another, you can query it to tell you whether there has been a transformation; e.g. resize, translation, rotation, etc. This is achievable because of the element mapping. If, for instance, (1, 1) is mapped onto (2, 2) in another sample, (2, 2) is mapped onto (3, 3), it doesn't take a genius to see that the transformation here is simply adding one to both x, and y values of the elements. That said, here is how I generalized the transformation

$$ \begin{bmatrix} \beta_1 & \beta_2 \\ \beta_3 & \beta_4 \\ \end{bmatrix} \begin{bmatrix} x_1 \\ y_1 \\ \end{bmatrix} + \begin{bmatrix} \beta_{5} \\ \beta_{6} \\ \end{bmatrix} = \begin{bmatrix} x_2 \\ y_2 \\ \end{bmatrix} $$

In the above equation $x_1$ and $y_1$ correspond to the element in the original sample, while $x_2$ and $y_2$ correspond to the value in the other sample where they were mapped. In the comp.row_match shown above, for instance, one of the matches is (7, 12) : (15, 16). That means element (7, 12) in sample 1 was mapped onto element (15, 16) in sample 2. In that regard, the above equation would transform into;

$$ \begin{bmatrix} \beta_1 & \beta_2 \\ \beta_3 & \beta_4 \\ \end{bmatrix} \begin{bmatrix} 7 \\ 12 \\ \end{bmatrix} + \begin{bmatrix} \beta_{5} \\ \beta_{6} \\ \end{bmatrix} = \begin{bmatrix} 15 \\ 16 \\ \end{bmatrix} $$

To solve this equation, I used the following steps;

$$ \begin{bmatrix} \beta_1x_1 & + & \beta_2y_1 \\ \beta_3x_1 & + & \beta_4y_1 \\ \end{bmatrix} + \begin{bmatrix} \beta_5 \\ \beta_6 \\ \end{bmatrix} = \begin{bmatrix} x_2 \\ y_2 \\ \end{bmatrix} $$

Then as follows;

$\beta_1x_1$ + $\beta_2y_1$ + $\beta_5$ = $x_2$
$\beta_3x_1$ + $\beta_4y_1$ + $\beta_6$ = $y_2$

These are two separate simple linear regression problems that I could rewrite as follows for sanity's sake

$\beta_1x_1$ + $\beta_2y_1$ + $\beta_0$ = $x_2$
$\beta_1x_1$ + $\beta_2y_1$ + $\beta_0$ = $y_2$

I solved the two linear regression to determine the transformation that occurred. In doing this, if you pick an image (or a sample in our lingo) and rotate it, doing comparison and getting the transformation matrix is expected to return the values indicating the transformation that occurred. You could also specify regions in the object based on how much of a match so we could know which parts of the image were rotated by how much

The solutions above can be interpreted the normal way. For instance, we know that a tranformation $\begin{bmatrix} 1 & 0 \\ 0 & 1 \\ \end{bmatrix}$ leaves the original element as is.
